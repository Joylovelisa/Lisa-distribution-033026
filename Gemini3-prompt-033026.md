WOW 醫療器材配送與流向分析工作室（Hugging Face Spaces / Streamlit）— 全面升級版綜合技術規格書
文件概述與系統定位
本文件為「WOW 醫療器材配送與流向分析工作室（Medical Device Distribution Analysis Studio）」的完整更新版技術規格書（Technical Specification）。本次升級旨在保留原有強大功能的基礎上，進一步擴展資料匯入彈性、資料集預覽機制、高階 Agentic AI 深度洞察能力、進階互動式 WOW 視覺化圖表，以及智慧化的 agents.yaml 設定檔管理機制。
系統的核心使命是賦能醫療器材供應鏈分析師、合規稽核人員與決策者，透過無程式碼（No-code）的直覺式 Web 介面（基於 Streamlit 架構），迅速將龐雜的配送紀錄轉換為具備商業價值的智慧洞察。本規格書詳盡定義了前端 UI/UX、後端資料處理管線、資料模型標準化策略、LLM（大型語言模型）代理服務路由機制，以及互動視覺化的具體規格，總字數與涵蓋深度皆符合企業級軟體開發與架構設計之標準。
1. 系統目標與核心功能全景 (System Objectives & Core Features)
1.1 既有核心功能保留與整合 (Legacy Features Retention)
系統將完全保留前一版本的所有核心功能：
Canonical Schema 轉換引擎：自動將非標準資料對齊至系統標準資料模型。
WOW UI 沉浸式主題系統：支援亮/暗模式、雙語切換（繁中/英文），以及 20 種藝術家風格（Painter Styles）與隨機 Jackpot 切換功能。
A/B 資料集深度比較模組：支援兩份資料集的獨立篩選、5 張對比圖表與 Markdown 差異摘要。
基礎 11 種 WOW 視覺化：包含 Sankey 流向圖、時間序列趨勢圖、Sunburst 層級圖、Heatmap 熱圖、Pareto 柏拉圖、Treemap 等。
靈活的部署與狀態監控：支援 Hugging Face Spaces 部署，頂部狀態列即時顯示 Token 消耗、API 狀態與資料列數。
1.2 全新升級功能模組 (New Features Specification)
本次迭代重點加入以下六大關鍵能力：
全方位資料集匯入引擎：支援使用者透過「上傳檔案（Upload）」或「純文字貼上（Paste）」匯入 TXT、CSV、JSON 格式的醫療器材配送資料。
動態彈性資料預覽機制：預設顯示前 20 筆標準化前/後的紀錄，並提供動態滑桿（Slider）或數字輸入框，讓使用者自由設定預覽筆數（例如 5 到 1000 筆）。
Agentic AI 深度長篇摘要報告：驅動 AI Agent 針對當前篩選之資料集，自動產出一份字數高達 2000~3000 字的 Markdown 格式「全景醫療器材配送洞察報告」。
5 款全新互動式 WOW 視覺化（含配送網路拓撲圖）：新增基於 Python (Plotly/NetworkX) 渲染的高階圖表。其中核心為「供應商 > 許可證字號 > 型號 > 客戶」的多節點網路圖，支援節點點擊互動與 7 大維度篩選器。
動態時間關聯互動資訊圖（Dynamic Time-Relation Infographic）：專注於分析「經銷商數量」、「設備數量」與「客戶數量」隨時間推移的動態交互關係。
Agent 工作流與智能 agents.yaml 治理：支援使用者與 Agent 的連續對話（Continuous Prompting），並提供 agents.yaml 的上傳/貼上/下載。內建 Schema 驗證與「LLM 強制標準化」功能，自動修復格式錯誤的 YAML 設定檔。
2. 核心資料架構與預設資料集 (Data Architecture & Canonical Schema)
2.1 標準化資料模型 (Canonical Schema Definition)
無論使用者匯入何種格式，系統將透過內部 Data Pipeline 將所有資料正規化（Normalize）為以下標準欄位，以便後續圖表與 Agent 呼叫：
supplier_id (字串)：供應商/經銷商代碼。
deliver_date (日期時間)：出貨或交貨日期（需轉換為 ISO 8601 或 pandas Timestamp）。
customer_id (字串)：接收端/客戶代碼。
license_no (字串)：衛福部醫療器材許可證字號。
category (字串)：醫療器材分類或品項類別（例如：E.3610、L.5980）。
udi_di (字串)：醫療器材單一識別系統之產品識別碼。
device_name (字串)：品名或裝置名稱。
lot_no (字串)：批號（Lot Number）。
serial_no (字串)：序號（Serial Number）。
model (字串)：產品型號。
quantity (整數)：配送數量（預設對應原始資料之 Number 欄位）。
2.2 預設資料集解析與載入策略
系統內建醫療器材配送樣本資料（包含 Boston Scientific 波士頓科技與 Neomedic 尼奧麥迪克等產品）。
解析邏輯：原始資料中的 Deliverdate (如 20251107) 將被解析為 2025-11-07；Category 與 DeviceNAME 中的全形引號將被清理；空白欄位（如某些紀錄無 SerNo）將補值為 NaN 或空字串。
狀態綁定：預設資料集將作為應用程式啟動時的 Base Session State，使用者隨時可點擊「重置為預設資料」按鈕復原。
3. 資料匯入與預覽工作室 (Data Ingestion & Preview Studio)
3.1 多元檔案匯入介面 (Multi-format Upload & Paste)
Upload (上傳檔案)：使用 st.file_uploader，支援 .csv, .json, .txt 副檔名。
CSV/TXT 解析：系統透過 Sniffer 自動偵測分隔符號（逗號、Tab、分號），並嘗試解碼（UTF-8, Big5, CP950）。
JSON 解析：支援 Array of Objects (Records) 以及包含中繼資料的嵌套 JSON。
Paste (文字貼上)：提供大型 st.text_area，使用者可直接從 Excel 或資料庫複製貼上。系統將觸發字串流解析器，利用 io.StringIO 將其轉換為 Pandas DataFrame。
3.2 動態資料預覽機制 (Dynamic Dataset Preview)
為了解決龐大資料集（如數十萬筆）在瀏覽器渲染卡頓的問題，同時滿足使用者檢核資料的需求，系統提供進階預覽功能：
預覽控制面板：提供一個滑桿（Slider）與數字輸入框（Number Input）的連動元件，標籤為「選擇預覽筆數 (Number of records to preview)」。
預設值與限制：預設值為 20 筆。最小值為 1，最大值可設定為 5000（考量前端記憶體限制）。
雙重視圖 (Dual View)：
Raw Data View：展示使用者匯入時的原始欄位名稱與未清洗的資料。
Standardized View：展示經過 Canonical Schema 映射後的乾淨資料，高亮顯示推斷映射的欄位（例如：原始 Number 被映射為 quantity）。
4. 全新 5 大 WOW 互動視覺化圖表 (The 5 New WOW Infographics)
在既有 11 種視覺化之上，系統新增專為醫療器材溯源與趨勢設計的 5 大 Python 渲染圖表。所有圖表皆具備高階互動性與 RWD 響應式佈局。
4.1 醫療器材配送網路拓撲圖 (Distribution Network Graph)
設計目標：以視覺化展現供應鏈實體流動關係，呈現上游到下游的完整路徑。
層級架構：固定四層節點結構：SupplierID (第一層) -> License number (第二層) -> model (第三層) -> CustomerID (第四層)。
視覺編碼 (Visual Encoding)：
節點大小 (Node Size)：依據流經該節點的總數量 (quantity) 決定。
邊線寬度 (Edge Width)：依據兩節點間的配送數量加權計算。
全域篩選器連動 (Global Filters)：使用者可透過側邊欄或頂部控制面板設定篩選條件：分類編號 (Classification No)、供應商 ID (SupplierID)、許可證字號、型號、批號/序號、客戶 ID 與日期區間 (Date Zone)。網路圖將根據條件即時重新渲染。
進階互動 (Node Click & Reveal)：
使用者點擊圖表中的任意節點（例如點擊某個特定的 CustomerID 節點）。
系統擷取該節點屬性，觸發回呼函數（Callbacks），在圖表下方或側邊浮動面板中展開「節點詳細資訊 (Node Detailed Information)」。
詳細資訊包含：涉及該節點的原始交易明細表、關聯的 UDI/批號清單、佔比統計圓餅圖等。
4.2 動態時間關聯互動圖 (Dynamic Time-Relation Infographic)
設計目標：展現特定器材分類在一段時間內的生態圈演變。
操作條件：使用者必須強制選擇「分類編號 (Classification No)」與「日期區間 (Date Zone)」。
視覺呈現：
X 軸：時間維度（自動依據日期跨度決定粒度：日、週或月）。
Y 軸（多軸設計）：
左側 Y 軸：經銷商/供應商的獨特數量（Count Distinct SupplierID）。
右側 Y 軸：設備/型號的獨特數量（Count Distinct model）。
氣泡大小/背景長條圖：客戶端的獨特數量（Count Distinct CustomerID）或總交易數量。
互動性：包含時間播放軸（Play/Pause Slider），讓使用者觀看隨時間推移，參與該類醫療器材買賣的經銷商與醫院（客戶）數量是如何擴張或收縮的。
4.3 3D 流向散佈矩陣圖 (3D Flow Scatter Matrix)
設計目標：以立體空間檢視大量設備型號在不同供應商與客戶間的群聚效應。
規格：X軸為供應商群組，Y軸為客戶群組，Z軸為時間。利用點的顏色代表許可證字號，並支援 360 度旋轉與縮放。使用者可直觀發現哪些時段發生了密集的特定型號大量移轉。
4.4 設備合規性與效期地理模擬圖 (Compliance & Zone Heat Node Map)
設計目標：以分區或叢集（Clusters）模擬器材的分布集中度。
規格：雖然資料集無經緯度，系統將基於 CustomerID 的編碼規則（若可推測地域）或採用力導向（Force-directed）分佈，將具有相同 LicenseNo 的設備聚集。用於檢視特定批號（Lot No）是否過度集中於少數客戶端，輔助產品召回（Recall）分析。
4.5 雙向階層弦圖 (Bidirectional Hierarchy Chord Diagram)
設計目標：展示供應商與大型客戶之間錯綜複雜的相互依賴關係。
規格：外圈圓環按比例分配給各 Supplier 與 Customer，內部的弦（Chords）代表 quantity 的流向。顏色漸層顯示從供應商端出發至客戶端的流動，滑鼠懸停可顯示精確的 UDI 轉移總量。
5. Agentic AI 深度洞察與分析引擎 (Agentic AI Insight Engine)
系統引入多個由 LLM 驅動的 Agent，不僅可處理短暫的任務，更擁有巨量文本生成的規格與連續互動的能力。
5.1 2000~3000 字全景醫療器材配送洞察報告 (Comprehensive Markdown Summary)
觸發機制：在 Dashboard 中提供「一鍵生成深度報告 (Generate Comprehensive Report)」按鈕。
資料封裝：系統會先將當前篩選過後的資料集進行預處理，提取統計摘要（Statistical Summary）、Top 10 排行榜、異常峰值（Z-score > 3 的交易）、以及唯一值計數，並將這些高密度中繼資料打包為 JSON 格式，送入 AI Agent 的 Context 中，避免傳輸原始全量資料導致 Token 溢出。
內容架構規範 (生成要求)：Agent 必須產出 2000 到 3000 字（包含豐富的 Markdown 排版）的報告，至少包含以下章節：
高階執行摘要 (Executive Summary)：300字的整體營運狀況概述。
供應鏈網絡集中度分析 (Supply Chain Concentration Analysis)：針對供應商與客戶的依賴關係進行 HHI（赫芬達爾—赫希曼指數）評估。
核心設備與許可證分佈 (Core Devices & License Distribution)：針對佔比最高的型號與許可證字號進行生命週期與流通廣度分析。
時間序列與異常節奏洞察 (Temporal Rhythm & Anomaly Insights)：基於日期維度的出貨波峰/波谷分析，點出不尋常的批號大量轉移事件。
合規與風險預警 (Compliance & Risk Assessment)：針對單一識別碼（UDI_DI）、批號連續性與潛在重複配送風險提出稽核建議。
行動決策建議 (Actionable Recommendations)：提供 3-5 點基於數據的具體商業與合規行動方針。
5.2 資料集連續提示對話與模型路由 (Continuous Prompting & Model Routing)
連續對話機制：使用者可以在生成上述報告後，繼續在 UI 介面下方輸入 Prompt（如：「請針對剛剛報告中提到的『C06030』客戶，進一步分析其接收的波士頓科技產品型號分佈」）。
指定範圍 (Select Models)：使用者可以透過下拉選單，選定特定的一組產品型號（Models），讓 Agent 只針對這些特定型號進行深度下鑽對話。
Agent 切換執行：使用者可隨時從左側的 Agent 清單（由 agents.yaml 解析而來）選定特定的 Agent 角色（例如切換至 data_quality_audit 進行資料品質盤點，或切換至 kpi_definition_catalog 生成新指標），並動態指定底層推論模型（如 OpenAI gpt-4o, Anthropic claude-3.5-sonnet, Gemini 1.5 Pro 等）。
狀態保存：系統利用 Streamlit Session State 儲存對話歷史與資料脈絡（Data Context），確保 Agent 能理解上下文。
6. Config Studio: 智慧型 agents.yaml 治理 (Intelligent Agents.yaml Management)
agents.yaml 是整個系統大腦運作的配置核心。為賦予使用者最大的調度彈性，系統提供完整的 YAML 檔案生命週期管理，並加入業界首創的「智能標準化機制」。
6.1 agents.yaml 的匯入與匯出 (Upload, Paste & Download)
匯出 (Download)：隨時可將系統當前記憶體中的 Agent 配置下載為標準 agents.yaml。
匯入 (Upload/Paste)：使用者可透過上傳 YAML 檔案或貼上 YAML 文字來更新 Agent 庫。
6.2 預設 agents.yaml 規格定義
依據使用者提供的預設檔，一個標準的 Agent 包含以下鍵值：
id：唯一識別碼（如 intake_goal_clarifier）。
name：顯示名稱。
description：用途描述。
provider：API 供應商（如 openai, gemini, anthropic）。
model：具體模型名稱（如 gpt-4o-mini）。
temperature：推論溫度參數。
max_tokens：最大生成字數。
system_prompt：系統級角色設定與輸出規範。
user_prompt：任務指示與輸入預期。
6.3 智慧型 YAML 結構轉換與標準化 (Agentic YAML Standardization)
當使用者上傳/貼上的 YAML 檔案不符合系統標準（例如：鍵值命名錯誤、縮排混亂、缺少必填欄位、格式為舊版 JSON 等），系統將啟動兩階段修復機制：
第一階段：啟發式規則修復 (Heuristic Parser & Schema Validation)
使用 Pydantic 進行 Schema 驗證。
自動補齊預設值（如缺少 temperature 預設補 0.2；缺少 provider 則依據 model 名稱如 gpt- 自動推斷為 openai）。
修正欄位名稱拼寫錯誤（例如將 sys_prompt 映射為 system_prompt）。
第二階段：LLM 強制標準化轉譯 (LLM Fallback Transformation)
若第一階段解析出現嚴重例外錯誤（Fatal Exception），系統會擷取該非標準的文字內容。
系統在背景發送一個特殊 Prompt 給預設的輕量化模型（如 GPT-4o-mini 或 Gemini-1.5-Flash）：「你是一個 YAML 架構工程師。請將以下非標準的文本轉換為嚴格遵循系統規格的 YAML 清單，必須包含 id, name, provider, model... 不得輸出任何其他解釋文字」。
接收 LLM 的輸出，重新驗證並套用於系統，並在 UI 上顯示「⚠️ 已透過 AI 將您的設定檔標準化成功」的提示與修復前後差異比較（Diff View）。
7. 部署與安全性架構 (Deployment, Security & Architecture)
7.1 前後端架構
Web 框架：Streamlit（單體架構 Single-file / Multi-page abstraction）。
資料操作：Pandas, Numpy。
視覺化渲染：Plotly Graph Objects, Plotly Express, NetworkX (供拓撲運算)。若要實作網路圖節點點擊事件，需整合 streamlit-plotly-events。
環境：完全相容並建議部署於 Hugging Face Spaces (Docker 或 Streamlit template)。
7.2 API 金鑰與敏感資料防護
憑證管理：支援從環境變數（.env 或 HF Space Secrets）自動注入 OpenAI/Anthropic/Gemini 金鑰。若無預設，則透過 UI 的 Password 型態輸入框請求，僅存於記憶體 Session 中。
資料隱私處理：若匯入資料包含敏感病患或院區資料（在醫療器材中，有時 CustomerID 或 UDI 具高機密性），系統在打包 JSON 送往 LLM 產生報告前，將執行本機端雜湊或遮蔽（Data Masking）。例如將原始 CustomerID (C05278) 替換為 Customer_A，並在 LLM 返回報告後，若有需要再進行本機端反向映射，確保絕對的資料合規性（HIPAA/GDPR 精神）。
7.3 巨量資料效能降載策略 (Performance Decimation Strategy)
針對超過 5 萬筆以上的配送紀錄：
視覺化降載：Sankey 圖與 Network 圖將自動啟用 Top-N 剪枝演算法。僅顯示流向數量佔總體 80% (Pareto) 的主幹節點，其餘小宗交易自動歸類為 Others 節點，防止瀏覽器 DOM 崩潰。
快取機制：全面使用 @st.cache_data 快取資料標準化結果與長耗時的 GroupBy 聚合運算。以篩選器條件的 Hash 值作為 Cache Key。
8. 全方位深度追蹤問題清單 (Comprehensive Follow-up Questions)
為確保本技術規格書在進入開發階段前能與利害關係人（Stakeholders）達成完美共識，以下列出 18 個需進一步釐清的架構、業務邏輯與使用者體驗問題：
關於資料匯入與預覽 (Data Ingestion & Preview)
預覽模式效能：使用者自訂預覽筆數最高可以設到多少？是否需要強制設定一個硬性上限（如 2000 筆）以免瀏覽器當機？
多重檔案追加：在匯入模式中，如果使用者先上傳了 File A，接著再上傳 File B，系統應該是「覆蓋」前一份資料，還是將兩者「合併（Append）」處理？
異常值判定：在標準化過程若發現日期格式錯亂或數量為負數，預覽畫面中是否需要以特別顏色（如紅色背景）高亮這些髒資料（Dirty Data）？
關於 Agentic 報告與連續對話 (Agentic AI)
報告生成時間：2000~3000 字的深度報告生成可能需要 30 秒至數分鐘，是否需要在畫面上提供逐字輸出的打字機效果（Streaming out）與進度條，以降低使用者等待焦慮？
指定型號對話：使用者選擇特定模型（Models）進行連續對話時，系統傳遞給 AI 的上下文（Context）是該型號的「逐筆明細」，還是僅傳遞「統計彙整」以節省 Token？
Token 限制與截斷：如果使用者選擇分析的資料量過大，超過了模型的 Context Window，我們應採用「隨機抽樣」、「擷取 Top N」還是「截斷歷史對話」策略？
報告匯出格式：生成的 3000 字 Markdown 報告，除了在畫面上檢視，是否需要提供匯出為 PDF 或 DOCX 檔案的功能？
關於 5 大 WOW 視覺化圖表 (Advanced Visualizations)
網路拓撲圖事件觸發：當點擊網路節點顯示詳細資訊時，若使用者點擊的是最上游的 Supplier 節點，展開的明細是否會過於龐大？是否需要針對明細表預設加入分頁（Pagination）機制？
動態時間關聯圖的粒度：時間動態圖表的 X 軸粒度（日、週、月），是希望由系統根據日期區間（Date Zone）自動切換，還是開放給使用者手動選擇？
圖表匯出：這些包含動態播放與點擊互動的 WOW 圖表，是否需要提供靜態 HTML 檔下載，讓使用者可以將可互動的圖表直接夾帶給外部客戶？
佈局配置：新增了 5 張高階圖表後，儀表板頁面將非常長。是否考慮引入頁籤（Tabs）或瀑布流模組化展開的設計來分類圖表？
關於 agents.yaml 智慧治理 (Config Studio)
LLM 修復的容錯性：當系統透過 LLM 將不標準的 agents.yaml 轉換為標準格式時，若 LLM 出現幻覺（Hallucination）自創了不存在的 provider，系統在儲存前是否要做最後一道硬性白名單（Whitelist）檢查？
預設設定檔覆寫：使用者下載修改後的 YAML，下次上傳後，是否應該存入瀏覽器的 Local Storage 中，讓使用者下次開啟網頁時不需重新上傳？
Prompt 版控：對於 Agent 的 Prompt 修改，是否需要提供簡單的版本控制（例如復原到上一步的修改），以防使用者改壞 Prompt？
關於 A/B 比較與系統整體架構 (System & Comparison)
A/B 資料集的 AI 運用：目前 AI 摘要主要針對單一資料集，是否需要新增一個特定的 Agent 任務，專門用來讀取 A、B 兩個資料集的統計特徵，並產出「A/B 差異深度診斷 AI 報告」？
主題與畫家風格相容性：WOW UI 的 20 種畫家風格（Painter styles）如何映射到新加入的網路圖與 3D 圖表的色彩配置（Colorscale）？是否需定義一套全局的 CSS to Plotly JSON 轉換腳本？
多語系覆蓋範圍：系統需支援繁體中文與英文切換，AI Agent 產出的 3000 字報告語言，是跟隨當前 UI 語系自動切換，還是在 Agent 的 prompt 裡提供手動選擇下拉選單？
預設資料集分離：在目前的指示中，預設的醫療資料與預設的 YAML 都寫在規格或 Prompt 中。進入實際開發時，是否同意將這些硬編碼（Hard-code）資料抽出，建立獨立的 default_data.csv 與 default_agents.yaml 檔案置於根目錄，以確保程式碼潔淨？default dataset:SupplierID,Deliverdate,CustomerID,LicenseNo,Category,UDID,DeviceNAME,LotNO,SerNo,Model,Number
